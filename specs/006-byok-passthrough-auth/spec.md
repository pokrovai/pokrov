# Спецификация фичи: BYOK Passthrough для шлюза

**Ветка фичи**: `006-byok-passthrough-auth`  
**Дата создания**: 2026-04-04  
**Статус**: Draft  
**Вход**: Описание пользователя: "Для шлюза с множеством клиентов нужна модель BYOK passthrough (клиентский ключ проходит через прокси), а не один OPENAI_API_KEY на инстанс. Добавить режимы upstream_auth_mode static|passthrough, отделить auth клиента к Pokrov от upstream Authorization, сохранить metadata-only аудит, выбирать профиль санитизации и rate limit по identity клиента."

## Пользовательские сценарии и тестирование *(обязательно)*

### Пользовательская история 1 - Прозрачный BYOK вызов LLM (Приоритет: P1)

Как клиент платформы, я отправляю запрос к LLM через Pokrov с собственным ключом провайдера и получаю ответ, при этом Pokrov сохраняет централизованные политики санитизации и контроля.

**Почему этот приоритет**: Это основной сценарий многоклиентного шлюза. Без него клиенты не могут использовать свои ключи провайдера через единый proxy.

**Независимая проверка**: Один клиент с валидным ключом провайдера успешно проходит через Pokrov в режиме passthrough; запрос и ответ обрабатываются политиками Pokrov.

**Сценарии приемки**:

1. **Given** включен passthrough-режим и клиент передает валидный provider-ключ, **When** клиент вызывает LLM endpoint через Pokrov, **Then** запрос обрабатывается и возвращается успешный ответ от upstream.
2. **Given** включен passthrough-режим, **When** клиент вызывает LLM endpoint без provider-ключа, **Then** Pokrov возвращает структурированную ошибку авторизации без обращения к upstream.

---

### Пользовательская история 2 - Разделение клиентской и upstream авторизации (Приоритет: P2)

Как оператор платформы, я хочу отделить доступ клиента к Pokrov от его provider-ключа, чтобы управлять доступом к шлюзу независимо от внешних ключей моделей.

**Почему этот приоритет**: Разделение доверенных контуров снижает риск смешения ролей аутентификации и позволяет управлять доступом по tenant/client policy.

**Независимая проверка**: Клиент проходит проверку доступа к Pokrov отдельным механизмом, а upstream-авторизация использует отдельные данные.

**Сценарии приемки**:

1. **Given** клиент передает корректные данные доступа к Pokrov, но неверный provider-ключ, **When** выполняется запрос, **Then** доступ к Pokrov подтверждается, но upstream-вызов завершается корректной ошибкой провайдера.
2. **Given** клиент не прошел проверку доступа к Pokrov, **When** выполняется запрос даже с валидным provider-ключом, **Then** запрос отклоняется на границе Pokrov.

---

### Пользовательская история 3 - Изоляция политик и лимитов по клиенту (Приоритет: P3)

Как оператор платформы, я хочу применять policy profile и rate-limit по клиентской идентичности, чтобы поведение шлюза было детерминированным и изолированным между клиентами.

**Почему этот приоритет**: Многоклиентный шлюз требует предсказуемой и справедливой изоляции политик и лимитов между tenant/client.

**Независимая проверка**: Два клиента с разными идентичностями получают разные policy/rate-limit решения без взаимного влияния.

**Сценарии приемки**:

1. **Given** два клиента с разными identity и разными профилями, **When** они отправляют сопоставимые запросы, **Then** каждый запрос оценивается и ограничивается по правилам своего клиента.
2. **Given** один клиент превышает свой лимит, **When** второй клиент в пределах лимита отправляет запрос, **Then** запрос второго клиента не блокируется из-за состояния первого.

---

### Edge Cases

- Что происходит, если в passthrough-режиме одновременно переданы client auth данные и provider auth данные в несовместимом формате?
- Как система ведет себя при upstream `401/403`, когда доступ к Pokrov успешен?
- Что происходит, если identity клиента отсутствует или не может быть разрешена в профиль политики?
- Как система ведет себя, если один и тот же provider-ключ ошибочно используется разными клиентами?

## Требования *(обязательно)*

### Функциональные требования

- **FR-001**: Система MUST поддерживать два режима upstream авторизации: `static` и `passthrough`.
- **FR-002**: В режиме `static` система MUST использовать заранее сконфигурированные provider credentials для upstream-вызовов.
- **FR-003**: В режиме `passthrough` система MUST использовать provider credentials, переданные клиентом в запросе, и MUST NOT подменять их глобальным ключом инстанса.
- **FR-004**: Система MUST разделять проверку доступа клиента к Pokrov и проверку provider credentials для upstream.
- **FR-005**: Система MUST отклонять запросы без обязательных данных клиентской авторизации на границе Pokrov до upstream-вызова.
- **FR-006**: Система MUST возвращать структурированные ошибки для случаев отсутствия/некорректности provider credentials в passthrough-режиме.
- **FR-007**: Система MUST определять профиль санитизации на основе identity клиента, а не на основе одного runtime ключа.
- **FR-008**: Система MUST применять rate-limit на основе identity клиента и его профиля.
- **FR-009**: Система MUST сохранять metadata-only аудит и MUST NOT записывать raw provider/client credentials в логи, аудит и метрики.
- **FR-010**: Система MUST сохранять текущие гарантии санитизации и policy enforcement для LLM/MCP трафика независимо от режима авторизации.
- **FR-011**: Система MUST обеспечивать обратную совместимость существующего сценария с `static` режимом без обязательной миграции клиентов.
- **FR-012**: Система MUST документировать контракт клиентской авторизации и passthrough поведения для операторов и клиентов шлюза.

### Ключевые сущности *(добавляйте, если фича работает с данными)*

- **Client Identity**: Идентификатор клиента/тенанта, используемый для выбора policy profile, rate-limit профиля и изоляции аудита.
- **Upstream Auth Mode**: Режим выбора источника provider credentials (`static` или `passthrough`) для исходящих вызовов.
- **Client Auth Context**: Набор данных, подтверждающих право клиента использовать Pokrov gateway.
- **Provider Credential Context**: Набор данных авторизации к upstream провайдеру, предоставленный статически или клиентом.
- **Tenant Policy Binding**: Связка client identity с правилами санитизации и лимитами.

## Ограничения безопасности и приватности *(обязательно)*

- Клиентские и provider credentials считаются чувствительными данными и MUST NOT попадать в логи, аудит, метрики и диагностические сообщения в raw-виде.
- Любой отказ по авторизации MUST быть metadata-only и не раскрывать секреты, токены или их фрагменты.
- Политики `allow/mask/redact/block` MUST применяться до отправки данных upstream.
- Разделение auth-контуров MUST исключать ситуацию, где валидный provider-ключ автоматически дает доступ к Pokrov без отдельной проверки клиентского доступа.
- Переход к passthrough MUST сохранять существующие границы безопасности для LLM и MCP путей.

## Операционная готовность *(обязательно для runtime-изменений)*

- **Логи**: MUST фиксировать режим авторизации, client identity, policy/rate-limit outcome и upstream status без секретов.
- **Метрики**: MUST учитывать раздельно события `static` и `passthrough`, а также блокировки по auth/policy/rate-limit.
- **Health/Readiness**: MUST оставаться совместимыми с текущими probe-контрактами; readiness отражает готовность обработки auth режимов.
- **Документация/конфиг**: MUST быть обновлены runtime config docs, основной README и примеры конфигурации для multi-client gateway сценариев.

## Required Test Coverage *(обязательно)*

- **Unit**: Выбор режима авторизации, разрешение identity клиента, выбор policy/rate-limit профиля, формирование metadata-only audit событий.
- **Integration**: Happy path для `passthrough` и `static`; block path для отсутствия client auth и отсутствия provider credentials; изоляция двух клиентов.
- **Performance**: Проверка, что дополнительная auth/identity логика не выводит накладные расходы за целевой бюджет runtime.
- **Security**: Проверка отсутствия утечки raw credentials в ошибках, логах, метриках и аудит-событиях.

## Success Criteria *(обязательно)*

### Измеримые результаты

- **SC-001**: Не менее 95% валидных клиентских запросов в режиме passthrough завершаются успешной обработкой через Pokrov без ручной конфигурации per-client provider key на инстансе.
- **SC-002**: 100% запросов без корректной клиентской авторизации отклоняются на границе Pokrov до обращения к upstream.
- **SC-003**: 100% тестов безопасности подтверждают отсутствие raw client/provider credentials в логах, аудит-событиях и метриках.
- **SC-004**: Для двух и более клиентов подтверждается изоляция policy/rate-limit: действия одного клиента не изменяют решения для другого.

## Acceptance Evidence *(обязательно)*

- Набор интеграционных сценариев, доказывающих happy path в `static` и `passthrough` режимах.
- Набор интеграционных и security сценариев, доказывающих корректный block/error path по auth и provider credentials.
- Артефакты проверки metadata-only логирования/аудита, подтверждающие отсутствие raw sensitive payload и credential data.
- Обновленные примеры конфигурации и runbook с проверкой по командам runtime в CI/локальной валидации.

## Assumptions

- Внешний ingress (например, mTLS/JWT слой) может предоставлять подтвержденную identity клиента для Pokrov.
- Для обратной совместимости существующие инсталляции могут оставаться в `static` режиме без изменений клиентских интеграций.
- Клиенты, использующие passthrough, несут ответственность за валидность своих provider credentials.
- В рамках этой фичи не добавляются новые продуктовые модули вне текущего v1 scope (A2A, SIEM, RBAC, UI).
