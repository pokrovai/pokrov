# Спецификация фичи: Bootstrap Runtime

**Ветка фичи**: `001-bootstrap-runtime`  
**Дата создания**: 2026-04-03  
**Статус**: Draft  
**Вход**: Описание пользователя: "Создать базовый runtime-каркас Pokrov.AI с валидируемым YAML-конфигом, request_id, структурированными логами, health/readiness endpoint'ами и корректным lifecycle для self-hosted сервиса."

## Пользовательские сценарии и тестирование *(обязательно)*

### Пользовательская история 1 - Запуск сервиса с валидным конфигом (Приоритет: P1)

Как platform engineer, я хочу запускать сервис с проверяемым конфигом, чтобы runtime не принимал трафик в некорректном состоянии.

**Почему этот приоритет**: Без валидного старта и предсказуемого lifecycle остальные возможности v1 не могут быть безопасно добавлены.

**Независимая проверка**: Запустить сервис с валидным YAML-конфигом и убедиться, что процесс стартует, отвечает на `/health` и переходит в ready только после завершения инициализации.

**Сценарии приемки**:

1. **Given** валидный конфиг и доступные runtime-зависимости, **When** сервис стартует, **Then** `/health` возвращает успешный ответ, а `/ready` становится успешным после завершения инициализации.
2. **Given** невалидный конфиг, **When** сервис стартует, **Then** процесс не переходит в состояние ready и возвращает понятную ошибку конфигурации.

---

### Пользовательская история 2 - Проверка состояния сервиса (Приоритет: P2)

Как оператор, я хочу различать liveness и readiness, чтобы корректно включать сервис в routing и исключать его при деградации или остановке.

**Почему этот приоритет**: Операционная готовность обязательна по конституции и PRD для self-hosted поставки.

**Независимая проверка**: Обратиться к `/health` и `/ready` в штатном режиме, во время старта и во время graceful shutdown.

**Сценарии приемки**:

1. **Given** процесс жив, но инициализация еще не завершена, **When** оператор вызывает `/ready`, **Then** сервис возвращает не-ready статус без ложноположительного успеха.
2. **Given** сервис обрабатывает остановку, **When** оператор вызывает `/ready`, **Then** сервис перестает считаться ready до завершения shutdown.

---

### Пользовательская история 3 - Трассировка запросов и событий (Приоритет: P3)

Как инженер сопровождения, я хочу видеть единый `request_id` и структурированные логи, чтобы разбирать инциденты без доступа к сырому payload.

**Почему этот приоритет**: Наблюдаемость и безопасное логирование являются обязательными правилами продукта.

**Независимая проверка**: Отправить запрос к сервису и убедиться, что ответ и журналы содержат одинаковый `request_id`, а сами логи не содержат сырого содержимого запросов.

**Сценарии приемки**:

1. **Given** входящий HTTP-запрос, **When** сервис формирует ответ, **Then** ответ содержит `request_id`, связанный с записью в structured log.
2. **Given** обычный runtime-трафик, **When** журналирование включено, **Then** логи содержат только метаданные и не содержат raw payload.

### Edge Cases

- Что происходит при пустом или частично заполненном YAML-конфиге?
- Как система ведет себя, если shutdown инициирован во время активных запросов?
- Что происходит, если проверка readiness вызывается до завершения загрузки конфигурации?

## Требования *(обязательно)*

### Функциональные требования

- **FR-001**: Система MUST загружать runtime-конфигурацию из YAML-файла при старте сервиса.
- **FR-002**: Система MUST валидировать обязательные поля конфигурации до перехода в состояние ready.
- **FR-003**: Система MUST предоставлять endpoint `/health`, подтверждающий, что процесс жив.
- **FR-004**: Система MUST предоставлять endpoint `/ready`, который отражает готовность к приему трафика и не возвращает успешный статус до завершения инициализации.
- **FR-005**: Система MUST присваивать каждому входящему запросу `request_id` и возвращать его в ответе.
- **FR-006**: Система MUST писать structured logs с обязательными полями времени, уровня, компонента, действия и `request_id`.
- **FR-007**: Система MUST выполнять graceful shutdown: сначала переводить сервис в not-ready, затем завершать активные запросы и только после этого останавливать процесс.
- **FR-008**: Система MUST документировать минимальный self-hosted сценарий запуска сервиса в контейнере и локальном режиме.
- **FR-009**: Система MUST отклонять конфигурации, которые нарушают правила хранения секретов или обязательные runtime-поля.
- **FR-010**: Система MUST исключать raw request/response payload из логов bootstrap-слоя по умолчанию.
- **FR-011**: Минимальный self-hosted сценарий MUST включать формализованные pass/fail проверки: успешный старт (`/health=200`), выход в ready (`/ready=200` после инициализации), корректный отказ (`/ready!=200`) до инициализации/во время draining.

### Ключевые сущности *(добавляйте, если фича работает с данными)*

- **RuntimeConfig**: Описание параметров запуска сервиса, включая сетевые настройки и ссылки на секреты.
- **RuntimeState**: Текущее состояние жизненного цикла сервиса: starting, ready, draining, stopped.
- **RequestContext**: Контекст обработки запроса с `request_id` и служебными метаданными для логирования.

## Ограничения безопасности и приватности *(обязательно)*

- Чувствительными считаются значения API key, будущие payload LLM/MCP и любые секреты из env/secret mounts; они MUST NOT попадать в логи bootstrap-слоя.
- Конфиг MUST использовать ссылки на секреты, а не открытые значения секретов в YAML.
- Readiness и lifecycle журналы MUST использовать только metadata-only поля без утечки входных тел запросов.

## Операционная готовность *(обязательно для runtime-изменений)*

- **Логи**: MUST появиться structured JSON-логи старта, изменения readiness, получения сигнала остановки и завершения shutdown с `request_id` для запросов.
- **Метрики**: MUST быть заложены базовые runtime hooks для последующего добавления Prometheus metrics без изменения публичного lifecycle-поведения.
- **Health/Readiness**: `/health` отражает liveness процесса, `/ready` отражает успешную загрузку конфига и готовность runtime.
- **Документация/конфиг**: MUST быть описаны обязательные поля YAML-конфига, ограничения на секреты и пример запуска.

## Required Test Coverage *(обязательно)*

- **Unit**: Валидация конфигурации, генерация/проброс `request_id`, переходы состояния lifecycle.
- **Integration**: Успешный старт, отказ при невалидном конфиге, поведение `/health` и `/ready`, graceful shutdown с активным запросом.
- **Performance**: Проверка, что bootstrap-слой не добавляет заметной деградации к обработке служебных endpoint'ов.
- **Security**: Проверка отсутствия raw payload в логах и отказа при некорректном задании секретов.

## Success Criteria *(обязательно)*

### Измеримые результаты

- **SC-001**: Сервис стартует с валидным конфигом и становится ready не позднее чем через 5 секунд на baseline-окружении.
- **SC-002**: 100% проверок `/ready` до завершения инициализации возвращают not-ready статус.
- **SC-003**: 100% ответов служебных endpoint'ов содержат `request_id`, пригодный для корреляции с журналами.
- **SC-004**: Проверка логов bootstrap-сценариев не выявляет ни одного случая записи raw payload или открытого секрета.
- **SC-005**: Quickstart-проверка self-hosted сценария считается успешной только при выполнении всех FR-011 pass/fail условий в локальном и контейнерном режимах.

## Acceptance Evidence *(обязательно)*

- Интеграционные тесты стартового пути, неготовности и graceful shutdown.
- Примеры structured logs с обязательными полями без чувствительных данных.
- Демонстрация локального и контейнерного запуска по quickstart.
- Подтверждение отказа старта на невалидном конфиге.

## Assumptions

- Реализация bootstrap runtime не включает бизнес-логику LLM- и MCP-проксирования.
- Health/readiness endpoints остаются неаутентифицированными, остальные runtime endpoint'ы могут требовать auth в следующих фичах.
- Базовый bootstrap-слой использует только in-memory состояние без внешнего хранилища.
- Контейнерный запуск является основным способом поставки, локальный запуск нужен для разработки и верификации.
- Baseline-окружение для acceptance: single-node Linux/amd64 или Darwin/arm64, не менее 2 vCPU и 4 GiB RAM, локальный loopback networking без внешнего LB.
