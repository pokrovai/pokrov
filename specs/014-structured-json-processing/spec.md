## Clarifications

### Session 2026-04-05

- Q: Какой явный latency-SLO зафиксировать для inline structured JSON processing? → A: P95 overhead sanitization+proxy ≤ 50 ms на representative nested payload.
- Q: Какой максимум входного JSON payload (до обработки) считаем поддерживаемым в рамках этой фичи? → A: До 1 MB.
- Q: Как система должна действовать, если structured JSON payload превышает 1 MB? → A: Пытаться обработать в best-effort режиме без SLA-гарантий.
- Q: Какой режим выбрать при внутренних ошибках structured обработки (например, сбой распознавания на листе)? → A: Fail-closed для high-risk контекста.
- Q: Какой уровень детализации пути разрешен в explain/audit summary? → A: Только path-safe категория/класс пути, без exact pointer.

## Пользовательские сценарии и тестирование *(обязательно)*

### Пользовательская история 1 - Детекция в структурированном JSON (Приоритет: P1)

Как оператор безопасности, я хочу, чтобы система детерминированно обходила вложенные JSON-объекты и массивы и анализировала только строковые листья, чтобы чувствительные данные обнаруживались без разрушения структуры полезной нагрузки.

**Почему этот приоритет**: Это базовая ценность фичи. Без корректного обхода и анализа строковых листьев последующие правила пути и трансформации не имеют смысла.

**Независимая проверка**: Подать вложенный JSON со смешанными типами и убедиться, что анализ применяется только к строкам, а форма JSON сохраняется.

**Сценарии приемки**:

1. **Given** JSON с вложенными объектами, массивами и строковыми листьями, **When** выполняется inline-анализ, **Then** каждая строка обрабатывается один раз с привязанным путем, а нестроковые значения остаются неизменными.
2. **Given** JSON с пустыми строками и смешанными типами, **When** выполняется обход, **Then** обход завершается без ошибок и пустые строки учитываются как валидные листья.

---

### Пользовательская история 2 - Политики по путям и полям (Приоритет: P2)

Как оператор политики, я хочу задавать правила включения/исключения семейств распознавателей и дефолтные действия трансформации по путям и алиасам полей, чтобы одинаковый текст в разных полях мог приводить к разным решениям.

**Почему этот приоритет**: Полевая семантика определяет качество контроля утечек в реальных полезных нагрузках, где контекст поля важнее буквального текста.

**Независимая проверка**: Подать JSON с одинаковыми строками в разных путях, настроить разные привязки и проверить, что решения различаются согласно приоритетам правил.

**Сценарии приемки**:

1. **Given** две одинаковые строки в разных путях и разные path-правила, **When** выполняется детекция и принятие решения, **Then** итоговые действия соответствуют правилам конкретного пути.
2. **Given** конфликтующие правила (точный путь, алиас, поддерево, профиль), **When** применяется движок правил, **Then** используется фиксированный порядок приоритетов и результат детерминирован.

---

### Пользовательская история 3 - Безопасные explain/audit summaries (Приоритет: P3)

Как инженер эксплуатации, я хочу получать объяснимые summary по структурированным данным без сырых значений, чтобы расследовать поведение политики без риска утечки чувствительных данных.

**Почему этот приоритет**: Операционная прозрачность нужна для верификации и расследований, но не должна нарушать privacy boundary.

**Независимая проверка**: Сгенерировать explain/audit summary после обработки вложенного JSON и подтвердить, что в выходе есть только безопасные счетчики и категории без raw-полей и raw-значений.

**Сценарии приемки**:

1. **Given** payload с чувствительными строками в нескольких путях, **When** формируется summary, **Then** summary содержит только path-safe метаданные, счетчики и категории без исходных значений.
2. **Given** обработка с блокирующими и неблокирующими результатами, **When** формируется аудит, **Then** аудит остается metadata-only и пригоден для операционной диагностики.

### Edge Cases

- Что происходит при глубоко вложенных структурах, содержащих пустые объекты, пустые массивы и пустые строки одновременно?
- Как система ведет себя при одинаковых строковых значениях в нескольких путях, для которых заданы взаимоисключающие path-правила?
- Что происходит, если вход содержит только нестроковые значения и ни одного строкового листа?
- Как система обрабатывает неизвестный путь, для которого нет явных path-привязок?
- Что происходит, если входной payload превышает 1 MB?
- Как система ведет себя при внутренней ошибке analyzer/transform на high-risk пути?

## Требования *(обязательно)*

### Функциональные требования

- **FR-001**: Система MUST выполнять детерминированный обход структурированного JSON по объектам и массивам, чтобы порядок обработки был воспроизводим между запусками.
- **FR-002**: Система MUST передавать в анализ только строковые листья и MUST сохранять все нестроковые значения без прямой трансформации.
- **FR-003**: Система MUST связывать каждый результат анализа строкового листа с безопасным контекстом пути и поля.
- **FR-004**: Система MUST поддерживать path-aware включение и исключение семейств распознавателей.
- **FR-005**: Система MUST поддерживать path-aware дефолтные действия трансформации и override для high-risk полей.
- **FR-006**: Система MUST применять фиксированный приоритет правил в порядке: точный путь, логический алиас поля, правило поддерева, профиль, глобальный дефолт.
- **FR-007**: Система MUST сохранять исходную форму JSON (структура объектов/массивов и позиционную семантику) после inline-трансформации.
- **FR-008**: Система MUST обеспечивать, что одинаковый текст в разных путях может приводить к разным решениям при разных path-привязках.
- **FR-009**: Система MUST формировать explain/audit summaries только в metadata-only формате: безопасные счетчики, категории и решения без raw-значений.
- **FR-010**: Система MUST использовать единый контракт hit/decision/transform для plain-text и structured JSON потоков.
- **FR-011**: Система MUST обрабатывать inline structured JSON payload размером до 1 MB в рамках целевого scope фичи с соблюдением заявленного latency-SLO.
- **FR-012**: Для payload больше 1 MB система MUST продолжать sanitization в best-effort режиме и MAY выходить за пределы latency-SLO, не нарушая security/privacy инварианты.
- **FR-013**: При внутренней ошибке structured обработки в high-risk контексте система MUST применять fail-closed решение и блокировать запрос.
- **FR-014**: Explain/audit summaries MUST включать только path-safe категории/классы пути и MUST NOT включать exact JSON pointer.

### Ключевые сущности *(добавляйте, если фича работает с данными)*

- **Structured Payload**: Вложенный JSON-документ, содержащий объекты, массивы и строковые/нестроковые листья.
- **String Leaf Context**: Набор безопасных атрибутов для строкового листа (путь, полевая категория, источник правил), используемый для политики и explain.
- **Path Binding Rule**: Политическое правило, связанное с точным путем, алиасом поля или поддеревом и влияющее на детекцию/трансформацию.
- **Structured Summary Record**: Metadata-only запись результатов обработки с агрегированными счетчиками, категориями и решениями.

## Ограничения безопасности и приватности *(обязательно)*

- Система MUST NOT включать raw-содержимое payload, значения строковых листьев или полные чувствительные пути в логи и аудит.
- Решения `allow`/`mask`/`redact`/`block` MUST применяться согласно profile- и path-привязкам без обхода приоритетов.
- Explain и audit MUST ограничиваться безопасными агрегатами и категориальными метками, достаточными для расследования без раскрытия данных.
- Ошибки обработки в high-risk контексте MUST приводить к блокировке запроса (fail-closed), а не к пропуску необработанных данных.
- Explain/audit MUST NOT содержать exact JSON pointer даже в штатной диагностике structured режима.

## Операционная готовность *(обязательно для runtime-изменений)*

- **Логи**: Должны появляться metadata-only события с request-id, итоговым policy decision и агрегированными счетчиками по structured-обработке.
- **Метрики**: Должны обновляться счетчики обработанных строковых листьев, срабатываний правил по путям и итоговых действий.
- **Health/Readiness**: `/health` и `/ready` остаются совместимыми; фича не должна ухудшать готовность сервиса при валидном конфиге.
- **Документация/конфиг**: Должны быть обновлены примеры profile/path bindings и инструкция верификации structured JSON сценариев.

## Required Test Coverage *(обязательно)*

- **Unit**: Детерминированный порядок обхода, корректность классификации string vs non-string leaf, применение приоритетов path-binding.
- **Integration**: Вложенный JSON с path-specific overrides, сценарии allow/mask/redact/block, сохранение формы payload после трансформации.
- **Performance**: Проверка, что на representative nested payload P95 суммарного overhead sanitization+proxy не превышает 50 ms.
- **Security**: Проверка metadata-only audit/explain и отсутствие raw leakage в логах/summary.

## Success Criteria *(обязательно)*

### Измеримые результаты

- **SC-001**: В 100% проверочных прогонов обработка вложенного JSON сохраняет исходную структуру и изменяет только строковые листья в соответствии с политикой.
- **SC-002**: Для заранее заданного набора конфликтующих path-правил система в 100% случаев применяет одинаковый и ожидаемый порядок приоритетов.
- **SC-003**: В acceptance-наборе с чувствительными данными 0 случаев попадания raw-значений в explain/audit outputs.
- **SC-004**: Не менее 95% целевых structured payload сценариев проходят end-to-end проверку без ручной корректировки политик после первичной настройки.
- **SC-005**: На representative nested payload P95 суммарного overhead sanitization+proxy не превышает 50 ms.
- **SC-006**: Для payload >1 MB в 100% проверочных прогонов сохраняются security/privacy инварианты (без raw leakage), даже если latency-SLO не гарантируется.
- **SC-007**: В 100% проверочных прогонов explain/audit summaries не содержат exact JSON pointer и содержат только path-safe категории.

## Acceptance Evidence *(обязательно)*

- Отчет по тестам на вложенных JSON-сценариях (happy path) с подтверждением детерминированного обхода.
- Отчет по конфликтующим path-привязкам (block/error path и override path) с фиксацией итоговых решений.
- Артефакт проверки логов/audit/explain на отсутствие raw sensitive payload.
- Сводный отчет по unit/integration/performance/security проверкам structured JSON режима.

## Assumptions

- Structured JSON режим применяется к inline-запросам прокси и не включает batch/табличный режим.
- Базовые contracts analyzer/transform/explain/audit уже определены и совместимы с расширением для path-context.
- Операторы имеют возможность конфигурировать profile/path bindings до включения фичи в production-трафик.
- Нестрогие или неизвестные пути без явных override используют профильные или глобальные дефолты без нарушения policy safety.
